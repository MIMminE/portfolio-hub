# 포트폴리오 면접 질문 은행

이 문서는 공개 블로그 글에 전부 넣기에는 긴 질문을 따로 모아둔 학습 노트입니다. 목표는 프로젝트를 설명한 뒤 이어질 수 있는 질문을 미리 준비하는 것입니다.

## 공통 메시지

전체 포트폴리오의 방향은 “백엔드 개발자가 AI를 활용해 프론트엔드와 앱, 패키징까지 확장했다”가 아니라, **백엔드/시스템 개발 경험으로 업무 흐름과 상태 모델을 정의하고 AI를 구현 보조 도구로 활용해 제품 형태까지 확장했다**는 점입니다.

## DevPilot

| 예상 질문 | 준비할 답변 방향 |
| --- | --- |
| 왜 macOS 앱과 Python CLI를 분리했나요? | UI와 자동화 로직의 책임 분리. SwiftUI는 네이티브 경험, Python CLI는 Jira/Git/Slack/Codex 연동 담당. |
| Codex 결과를 그대로 믿지 않기 위한 장치는 무엇인가요? | AI 분석, repo 확정, workspace, 테스트, 보고를 단계로 나누고 사람이 승인/기록하게 설계. |
| 왜 로컬 우선 구조인가요? | Jira 내용, 토큰, 로컬 경로, 개인 메모가 포함되므로 외부 서버 없이 동작하는 구조가 적합. |

## Kotlin Commerce Core

| 예상 질문 | 준비할 답변 방향 |
| --- | --- |
| Outbox를 왜 썼나요? | DB 저장과 Kafka 발행을 하나의 분산 트랜잭션으로 묶을 수 없기 때문에, 저장 트랜잭션 안에서 발행 의도를 먼저 기록. |
| Lease만 있으면 중복 발행이 완전히 없어지나요? | 아니다. 경합 가능성을 줄이는 장치이고, consumer 멱등 처리가 함께 필요하다. |
| Idempotency-Key와 eventId 멱등 처리는 어떻게 다른가요? | Idempotency-Key는 API 재요청 방어, eventId는 이벤트 consumer 중복 처리 방어. |
| MSA의 핵심을 무엇으로 설명하나요? | 서비스 개수가 아니라, 분리 이후의 실패 모델과 정합성 보완 패턴. |

## Warehouse Ops Suite

| 예상 질문 | 준비할 답변 방향 |
| --- | --- |
| 입고 예정, 검수, 적치 수량을 왜 분리했나요? | 실제 물류 현장에서는 예정 수량과 검수/적치 결과가 다를 수 있고, 한 입고 라인이 여러 로케이션으로 나뉠 수 있기 때문. |
| 3PL 재고 모델에서 무엇이 중요한가요? | SKU뿐 아니라 고객사, 창고, 로케이션, 가용/할당 수량을 함께 봐야 한다. |
| DPS/Print Agent를 왜 분리했나요? | 현장 장비는 로컬 네트워크와 장비 상태에 의존하므로 중앙 API와 장애 영향 범위를 분리해야 한다. |
| 회사 경험을 어떻게 공개 가능한 포트폴리오로 만들었나요? | 회사 코드/데이터/명칭을 쓰지 않고 업무 흐름과 상태 모델만 일반화한 Clean-room 프로젝트로 재구성. |

## Stock Lock Benchmark

| 예상 질문 | 준비할 답변 방향 |
| --- | --- |
| 낙관락과 비관락 중 무엇이 더 좋나요? | 정답은 트래픽 패턴에 따라 다르다. 단일 row 핫스팟에서는 retry 비용 때문에 비관락이 더 예측 가능할 수 있다. |
| 평균 latency만 보면 안 되는 이유는 무엇인가요? | 충돌과 retry는 tail latency를 크게 키우므로 p95/p99와 retry 수를 함께 봐야 한다. |
| 실험의 한계는 무엇인가요? | 단일 row 핫스팟 조건이므로 일반적인 분산 SKU 트래픽에 그대로 일반화하면 안 된다. |
| 이 프로젝트가 보여주는 역량은 무엇인가요? | 동시성 전략을 감이 아니라 실험 조건, 지표, 한계로 설명하는 백엔드 설계 역량. |

## Care Commerce Platform

| 예상 질문 | 준비할 답변 방향 |
| --- | --- |
| client/admin API를 왜 분리했나요? | 같은 도메인 데이터라도 사용자와 관리자는 권한, 응답 모델, 감사 기준이 다르기 때문. |
| 동의 이력을 별도 모델로 둔 이유는 무엇인가요? | 약관/개인정보/민감정보/마케팅 동의는 현재값뿐 아니라 언제 어떤 항목에 동의했는지가 중요하기 때문. |
| 관리자 조회를 감사 로그로 남기는 이유는 무엇인가요? | 회원 상세나 케어 프로필 조회는 운영 리스크가 있는 개인정보 접근 이벤트이기 때문. |
| 주문 가격 스냅샷은 왜 필요한가요? | 상품 가격이 변경되어도 과거 주문 금액은 변경되면 안 되기 때문. |

## 답변 톤

- 구현 기술보다 먼저 문제와 의사결정을 설명한다.
- “AI가 만들어줬다”보다 “AI를 활용해 구현 범위를 넓혔고, 요구사항/상태/경계/검증 기준은 직접 정의했다”고 말한다.
- 모든 프로젝트에서 “왜 이 구조를 선택했는가”와 “어떤 한계가 있는가”를 함께 말한다.
