# DevPilot

DevPilot는 개발자의 하루 업무 흐름을 하나의 로컬 앱 안에서 이어주기 위해 만든 macOS 중심 개인 자동화 시스템입니다.

Jira 일감, 로컬 Git 저장소, Codex 작업 요청, Slack 보고, 빠른 메모, 연장근무 기록은 실제 개발 업무에서 계속 흩어집니다. DevPilot는 이 전환 비용을 줄이기 위해 오늘 해야 할 일을 확인하고, 작업을 시작하고, 근거를 남기고, 보고서와 기록으로 마무리하는 흐름을 하나의 메뉴바 앱으로 묶었습니다.

## 이 프로젝트에서 확인할 수 있는 역량

- Jira, Git, GitHub CLI, Slack, Codex CLI를 하나의 로컬 업무 흐름으로 연결한 자동화 설계
- SwiftUI macOS 앱과 Python CLI를 분리한 로컬 우선 애플리케이션 구조
- 브랜치 준비, 작업 컨텍스트 정리, 보고서 초안 생성 같은 반복 업무의 도구화
- 실제 업무 데이터를 공개하지 않기 위한 마스킹 모드와 공개 문서화 방식
- AI 도구를 단발성 코드 생성이 아니라 작업 지휘와 기록 관리 흐름으로 확장한 경험

## 프로젝트 목표

이 프로젝트에서 보여주고 싶은 것은 단순한 TODO 앱이 아닙니다. 개발자가 실제로 하루 동안 오가는 도구들을 제품 형태로 엮고, 그 과정에서 필요한 자동화와 기록 구조를 어떻게 설계했는지를 보여주는 것이 핵심입니다.

| 목표 | 설명 |
| --- | --- |
| 업무 맥락 통합 | Jira 일감, 저장소, 브랜치, 커밋, 보고서를 하나의 흐름으로 연결 |
| 반복 작업 자동화 | 브랜치 준비, Codex 요청 프롬프트, 일일 보고서 초안 생성 |
| 로컬 우선 설계 | 업무 토큰과 기록을 로컬 설정과 로컬 상태로 관리 |
| 공개 문서화 | 마스킹 모드와 화면 캡처로 포트폴리오 설명 가능 |

최근에는 이 목표를 `일감 하나의 생애주기`로 더 좁혔습니다.

```text
일감 수신
-> AI 1차 분석
-> repository 확정
-> issue workspace 준비
-> Codex와 구현
-> 테스트 기록
-> 보고/완료
```

이 흐름에서 AI는 코드 생성 버튼이 아니라 작업 사이사이에 들어오는 구현 파트너에 가깝습니다. 처음에는 요구사항을 As-Is/To-Be, 영향 범위, 리스크로 나누고, workspace가 만들어진 뒤에는 일감 상태와 repository, 분석 결과, 테스트/보고 기록이 포함된 컨텍스트를 기준으로 구현을 이어갑니다.

## 해결하고 싶었던 문제

DevPilot를 만들면서 중요하게 본 지점은 기능 개수가 아니라, 실제 개발자가 하루 동안 반복하는 전환 비용이었습니다.

| 기존 업무 흐름의 문제 | DevPilot에서의 대응 |
| --- | --- |
| Jira, Git, Codex, Slack, 메모가 각각 분리되어 작업 맥락이 끊김 | 일감 단위 workflow로 분석, repository, workspace, 테스트, 보고를 한 흐름에 보관 |
| Codex에게 매번 Jira 배경, 저장소 구조, 테스트 기준을 다시 설명해야 함 | 일감 요약, 연결 repository, `AGENTS.md`, 테스트 기대값을 작업 컨텍스트로 구성 |
| 여러 repository가 얽힌 작업에서 어느 저장소를 수정해야 하는지 놓치기 쉬움 | 관리 repository 상태와 일감 내용을 기준으로 후보 repository와 브랜치 전략을 제안 |
| AI가 만든 결과와 사람이 검증한 근거가 따로 흩어짐 | 테스트 결과, 보고 초안, 리스크를 일감 타임라인에 다시 기록 |
| 업무 보고를 위해 커밋, Jira, 메모를 다시 찾게 됨 | 하루 기록을 보고서 초안과 날짜별 히스토리로 재사용 |

## 대표 시나리오: 일감 하나를 끝까지 밀고 가기

예를 들어 `LMS-493 출고 검수 화면의 예외 상태 표시 개선` 같은 일감이 들어왔다고 가정합니다. 일반적인 업무 흐름에서는 Jira를 읽고, 관련 저장소를 떠올리고, 브랜치를 만들고, Codex에게 다시 설명하고, 테스트 결과를 별도로 메모한 뒤, 마지막에 Slack 보고 문장을 다시 작성해야 합니다.

DevPilot는 이 반복을 하나의 흐름으로 묶습니다.

![DevPilot 일감 처리 콘솔](./images/06-issue-flow-console.png)

| 단계 | 화면에서 확인하는 것 | 효과 |
| --- | --- | --- |
| 일감 수신 | 프로젝트별 일감, 승인 대기/진행/보고 상태 | 오늘 처리할 일을 Jira 탭을 계속 오가지 않고 선택 |
| AI 분석 | As-Is/To-Be, 영향 범위, 리스크, 확인 질문 | 구현 전에 요구사항의 빈틈과 위험을 먼저 정리 |
| repository 확정 | 변경 후보 저장소와 현재 Git 상태 | 프론트/백엔드/에이전트처럼 여러 저장소가 얽힌 작업 누락 방지 |
| workspace 준비 | 일감 전용 worktree와 브랜치 | 기존 작업 폴더와 섞이지 않는 격리된 작업 환경 |
| Codex 작업 | 일감 요약, 저장소, 컨벤션, 테스트 기준을 포함한 컨텍스트 | AI에게 매번 같은 배경 설명을 반복하지 않음 |
| 테스트/보고 | 테스트 결과, 보고 초안, 완료 상태 | 구현 근거가 기록과 보고서로 자연스럽게 이어짐 |

`AI 작업 지휘관`은 이 흐름 중간에 있는 판단 보드입니다. 단순히 “AI에게 물어보기” 버튼을 두는 것이 아니라, 현재 일감 상태를 바탕으로 작업 계획, 브랜치 전략, 테스트 추천, 컨벤션 점검, 보고 초안을 나눠 보여줍니다.

![DevPilot AI 작업 지휘관](./images/07-ai-director.png)

이 화면에서 중요한 점은 AI 결과를 그대로 믿게 만드는 것이 아니라, 사람이 승인하고 확인해야 할 지점을 드러내는 것입니다. 예를 들어 테스트 결과가 아직 없거나 보고가 등록되지 않았으면 리스크로 표시하고, 연결된 repository의 브랜치 상태가 맞는지 함께 확인하게 했습니다.

## Codex와 통신하는 방식

DevPilot는 Codex를 단발성 코드 생성 도구로만 사용하지 않습니다. 앱이 먼저 업무 맥락을 정리하고, Codex가 읽을 수 있는 작업 컨텍스트를 로컬 workspace에 만들어 둔 뒤, 같은 작업 루트에서 구현을 이어가게 합니다.

| 입력 맥락 | Codex에 전달되는 이유 |
| --- | --- |
| Jira/수동 일감 요약 | 구현 목표와 완료 조건을 분명히 하기 위해 |
| AI 1차 분석 결과 | As-Is/To-Be, 영향 범위, 리스크를 작업 전에 공유하기 위해 |
| repository 후보와 Git 상태 | 어떤 저장소를 수정해야 하는지, 현재 브랜치가 안전한지 확인하기 위해 |
| `AGENTS.md`와 로컬 컨벤션 | 프로젝트별 코드 스타일과 명령을 반복 입력하지 않기 위해 |
| 테스트/보고 기록 | 변경 후 검증과 업무 보고가 같은 흐름에 남도록 하기 위해 |

실제 Codex 대화 화면에는 개인 작업 기록이나 회사 업무 맥락이 섞일 수 있어 공개 이미지로 사용하지 않았습니다. 대신 DevPilot 화면에서는 어떤 일감에서 시작했고, 어떤 repository가 연결됐고, 어떤 검증과 보고가 남았는지를 샘플 데이터로 보여주도록 구성했습니다.

## 설계 판단

DevPilot는 개인 자동화 도구이지만, 구조는 실제 업무 도구처럼 오래 사용할 수 있는 방향을 의식했습니다.

| 설계 선택 | 이유 |
| --- | --- |
| SwiftUI 앱과 Python CLI 분리 | 화면은 macOS 네이티브 경험으로 만들고, Jira/Git/Slack/보고서 자동화는 CLI로 분리해 테스트와 확장을 쉽게 하기 위해 |
| 로컬 우선 저장 | 업무 토큰, Jira 내용, 개인 메모가 외부 서버에 불필요하게 올라가지 않도록 하기 위해 |
| 단계별 승인 흐름 | AI 분석과 자동화를 바로 실행하지 않고, 사람이 확인해야 하는 지점을 명확히 드러내기 위해 |
| 일감 단위 workspace | 여러 작업이 섞이지 않도록 worktree와 컨텍스트 파일을 일감 기준으로 격리하기 위해 |
| 마스킹 모드 | 실제 업무 데이터를 다루는 앱도 포트폴리오에서 안전하게 설명할 수 있도록 하기 위해 |

이 프로젝트의 핵심은 “AI로 앱을 만들었다”가 아니라, 백엔드/시스템 개발 경험을 바탕으로 업무 흐름을 먼저 정의하고, AI를 그 흐름 안에 넣어 macOS 앱, CLI, 외부 연동, 패키징까지 확장했다는 점입니다.

## 대시보드

![DevPilot 대시보드](./images/01-dashboard.png)

대시보드는 앱을 켰을 때 가장 먼저 보는 화면입니다. 오늘 할 일, 새 Jira 일감, 작업 중인 저장소, 정비가 필요한 저장소, Codex 상태, 보고서 상태를 요약합니다.

이 화면의 목적은 모든 정보를 펼쳐놓는 것이 아니라, 오늘 먼저 봐야 할 신호를 짧게 보여주는 것입니다. 개발자가 아침에 앱을 열었을 때 "어떤 일감을 시작해야 하는지", "어떤 저장소가 밀려 있는지", "보고서 작성 근거가 쌓이고 있는지"를 바로 판단할 수 있게 했습니다.

## 작업 화면

![DevPilot 작업 화면](./images/02-work.png)

작업 화면은 Jira 일감에서 실제 개발 작업으로 넘어가는 중심 흐름입니다.

```text
Jira 일감 선택
-> 기획 본문, 첨부, 댓글 확인
-> 관련 저장소 연결
-> Jira 키 기반 브랜치 준비
-> Codex 작업 요청 프롬프트 생성
-> 개발, 커밋, PR, 보고서
```

DevPilot에서 중요한 점은 "브랜치를 만든다" 같은 단일 명령이 아니라, 명령 주변의 맥락을 함께 보존하는 것입니다. 어떤 Jira 일감에서 시작했는지, 어떤 저장소가 연결됐는지, 어떤 브랜치가 만들어졌는지, Codex에게 어떤 맥락을 넘겼는지가 하나의 기록으로 이어집니다.

작업이 길어지는 경우에는 일감 처리 콘솔에서 이어갑니다. 콘솔은 `AI 분석`, `repo 확정`, `workspace 준비`, `업무 처리`, `테스트`, `보고/완료`를 단계로 보여주고, 각 단계마다 승인 버튼과 다음 행동을 제공합니다.

## 저장소 관리

![DevPilot 저장소 화면](./images/03-repositories.png)

저장소 화면은 관리 중인 로컬 Git repository 상태를 확인하는 공간입니다.

| 항목 | 설명 |
| --- | --- |
| 기준 브랜치 | 저장소별 작업 기준 브랜치 |
| 현재 브랜치 | checkout된 브랜치와 작업 브랜치 여부 |
| 동기화 상태 | ahead, behind, pull, rebase 필요 여부 |
| 오늘 작업 | 오늘 발생한 커밋과 머지 |
| PR/릴리즈 | GitHub에서 확인한 PR과 최신 릴리즈 |

하나의 Jira 일감이 여러 저장소에 걸쳐 있을 때, 각 저장소의 브랜치와 커밋 상태를 한 화면에서 비교할 수 있게 만든 점이 핵심입니다.

![DevPilot 보고서 화면](./images/04-report.png)

보고서 화면은 오늘의 작업 근거를 바탕으로 일일 보고서 초안을 만드는 곳입니다.

DevPilot는 오늘 커밋, 머지, Jira 작업 맥락, 메모, 보고서 규칙을 모아 초안을 만들고, 사용자가 그 초안을 ChatGPT나 Codex에 전달해 다듬을 수 있게 합니다. AI API가 없어도 사용할 수 있도록 "근거 수집과 프롬프트 생성"을 앱의 기본 기능으로 두었습니다.

## 기록

![DevPilot 기록 화면](./images/05-records.png)

기록 화면은 보고서, 메모, Jira 흐름, 연장근무 기록을 날짜별 타임라인으로 다시 확인하는 공간입니다.

개발 업무는 나중에 "그때 왜 그렇게 했는지"를 찾아야 할 때가 많습니다. DevPilot는 하루의 작업 근거를 보고서와 기록으로 보관해 개인 업무 히스토리로 활용할 수 있게 합니다.

## 아키텍처

DevPilot는 SwiftUI 앱과 Python CLI를 분리한 구조입니다.

```text
SwiftUI macOS 메뉴바 앱
-> DevPilotRunner 브리지
-> Python CLI: devpilot
-> Jira API / GitHub CLI / Slack API / 로컬 Git / 로컬 앱 상태
-> Codex CLI
-> GitHub Actions 태그 기반 macOS 릴리즈
```

| 계층 | 역할 |
| --- | --- |
| SwiftUI 앱 | macOS 네이티브 UI, 메뉴바 앱, 대시보드, 작업, 저장소, 보고서, 기록 |
| Python CLI | Jira, Git, Slack, 보고서, 연장근무 자동화 명령 |
| 로컬 상태 | 프로필, Jira-저장소 연결, 보고서, 메모, 연장근무 기록 |
| Codex CLI | AI 기반 작업 요청과 보고서 다듬기 |
| GitHub Actions | 태그 기반 macOS 앱 릴리즈 |

SwiftUI 앱은 사용자가 보는 제품 경험을 담당하고, Python CLI는 외부 API와 로컬 자동화를 담당합니다. 이렇게 나누면 앱 화면과 자동화 로직을 분리해 테스트하고 확장하기 쉽습니다.

## 공개 문서화를 위한 마스킹

DevPilot는 실제 업무 도구를 다루기 때문에 Jira 키, 저장소명, 사람 이름, 커밋 메시지, 로컬 경로가 노출될 수 있습니다. 그래서 공개용 캡처를 만들 때 사용할 수 있는 마스킹 모드를 두었습니다.

포트폴리오에 포함된 화면은 모두 마스킹 모드를 켠 상태로 촬영한 공개용 샘플입니다. 설정 화면처럼 이메일, 토큰, 로컬 경로가 직접 보일 수 있는 화면은 게시물에서 제외했습니다.

## 릴리즈 방식

DevPilot는 태그 기반 GitHub Release를 사용합니다.

```bash
git tag v0.1.x
git push origin v0.1.x
```

릴리즈 워크플로우는 Python CLI를 PyInstaller로 빌드하고, SwiftUI macOS 앱 번들을 조립한 뒤 `devpilot-macos-menubar-arm64.zip`을 GitHub Release에 첨부합니다.

## 포트폴리오 포인트

DevPilot는 특정 회사의 업무 데이터를 공개하지 않으면서도, 실제 개발자의 업무 흐름에서 생긴 문제를 제품으로 정리한 프로젝트입니다.

- SwiftUI 기반 macOS 네이티브 앱
- Python CLI 기반 자동화 백엔드
- Jira, GitHub, Slack, Git, Codex CLI 연동
- 업무용/개인용 프로필 분리
- 반복 가능한 Codex 작업 요청을 위한 구조화된 프롬프트 생성
- 날짜별 업무 기록과 보고서 저장
- 공개 문서화를 위한 마스킹 모드
- GitHub Actions 기반 macOS 앱 릴리즈

이 프로젝트를 이력서나 면접에서 설명할 때는 “프론트엔드 앱을 만들었다”보다 “기존 백엔드/시스템 개발 경험을 바탕으로 업무 흐름을 정의하고, AI를 활용해 macOS 앱, CLI, 외부 연동, 패키징까지 제품 형태로 확장했다”는 점을 강조하는 편이 더 좋습니다.

## 회고

DevPilot를 만들면서 가장 중요하게 본 것은 "자동화 스크립트를 많이 만든다"가 아니라 "업무 흐름이 끊기지 않게 만든다"였습니다.

개발자는 Jira, Git, Slack, AI 도구, 로컬 저장소 사이를 계속 이동합니다. DevPilot는 그 이동을 하나의 제품 경험으로 묶고, 하루가 끝났을 때 보고서와 기록으로 남기는 구조를 실험한 프로젝트입니다.
